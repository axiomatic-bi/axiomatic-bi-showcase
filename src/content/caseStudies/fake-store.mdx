The objective is to prove a repeatable reporting workflow from API-first source data, without embedding business logic in dashboards.  
This example shows that ingestion, transformation, and BI delivery can be versioned independently while still producing one consistent analytical contract.

## Responsibility split

### Engine layer (ingestion and transformation)

- Source extraction and load policies are defined in [`projects/fake_store/run_pipeline.py`](https://github.com/axiomatic-bi/axiomatic-engine/blob/main/projects/fake_store/run_pipeline.py).
- Transformation contracts live in dbt model files and tests, including [`models/gold/schema.yml`](https://github.com/axiomatic-bi/axiomatic-engine/blob/main/projects/fake_store/dbt_project/models/gold/schema.yml).
- Runtime and local execution assumptions are documented in [`projects/fake_store/.env.example`](https://github.com/axiomatic-bi/axiomatic-engine/blob/main/projects/fake_store/.env.example).

### Reporting layer (semantic and BI outputs)

- The Power BI artefact consumes the same warehouse contract and is published separately from runtime code.
- Report outputs are implementation evidence, not a place for hidden pipeline logic.

## Modelling and transform decisions

### Model contract

Gold models follow a star-schema contract centred on cart-item facts with conformed user, product, and date dimensions.  
This keeps joins and grain explicit for downstream BI consumers.

### Load policy decisions

Resource-level load hints define rerun semantics at endpoint level (`merge` for entity updates, `replace` for snapshot-style refreshes).  
This improves operational clarity and supports deterministic behaviour.

### Validation approach

Quality checks are maintained in dbt schema tests and run in the same transformation path as model execution.  
This reduces drift between transform logic and validation logic.

## Outcomes and limitations

- The current example proves deterministic rerun expectations and reproducible outputs from documented commands.
- It provides a baseline for comparing future source adapters and front-end reporting consumers.
- It is demonstration-focused and does not include full production scheduling or alerting.
- Semantic MCP integration remains planned rather than implemented in this example.
